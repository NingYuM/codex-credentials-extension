# 预览区复制按钮设计

## 背景

当前插件在弹窗中提供一个主按钮“抓取并复制 auth.json”，它会抓取当前浏览器中的 `chatgpt.com` 会话、生成 `auth.json`、展示预览，并立即复制到剪贴板。

用户希望保留现有主按钮行为，同时在“预览 auth.json (含敏感信息)”文字后面新增一个单独的复制按钮，用于再次复制当前已经生成的预览内容，而不重新执行抓取。

## 目标

- 保留现有主按钮“抓取并复制 auth.json”的行为不变。
- 在预览区域标题后新增一个次级“复制”按钮。
- 该按钮只在预览区域显示后出现。
- 点击该按钮时只复制当前预览中的 `auth.json` 文本，不重新抓取数据。
- 复制成功后，通过状态区给出明确反馈。

## 非目标

- 不修改当前抓取逻辑。
- 不更改主按钮文案或职责。
- 不增加新的存储、缓存或后台通信逻辑。
- 不引入新的权限申请。

## 界面设计

### 预览标题区

当前预览区域使用 `details > summary` 展示标题文本。这里将调整为一个包含标题文本和次级复制按钮的标题行。

布局要求：
- 标题仍然表达“预览 auth.json (含敏感信息)”这一含义。
- 新按钮在视觉上紧跟标题，位于同一行末尾。
- 该按钮为次级操作，样式应弱于主按钮，避免与主按钮竞争。
- 预览区域隐藏时，新按钮也隐藏。

### 显示时机

- 初始状态：预览区域隐藏，复制按钮不显示。
- 抓取成功后：预览区域显示，复制按钮同时显示。
- 抓取失败时：保持预览区域隐藏，复制按钮不显示。

## 交互设计

### 主按钮

主按钮继续执行以下流程：
1. 请求当前浏览器会话。
2. 生成 `auth.json`。
3. 将内容写入预览区。
4. 自动复制到剪贴板。
5. 显示状态信息。
6. 显示预览区和新的复制按钮。

### 新增复制按钮

新增按钮执行以下流程：
1. 读取当前预览区中的文本内容。
2. 复用已有复制方法写入剪贴板。
3. 更新状态区，提示已复制当前预览内容。

该按钮不执行以下行为：
- 不重新调用抓取逻辑。
- 不重新请求 `chatgpt.com/api/auth/session`。
- 不修改当前预览内容。

## 实现设计

### `popup.html`

需要修改预览区域结构：
- 在预览区标题附近新增一个复制按钮节点。
- 为标题区增加可布局的容器，便于让标题和按钮并排显示。
- 为次级按钮增加必要的样式选择器。

### `popup.js`

需要扩展以下逻辑：
- 获取新增复制按钮的 DOM 引用。
- 在初始化时确保该按钮隐藏。
- 在抓取成功后显示该按钮。
- 在抓取失败或重新开始抓取时隐藏该按钮，避免用户操作旧状态。
- 为新增按钮绑定点击事件。
- 点击时读取 `previewContent` 的当前文本并调用已有的 `copyToClipboard()`。
- 复制成功后更新 `#status`，提示已复制预览内容。

### 复用策略

现有的 `copyToClipboard()` 已经封装了剪贴板复制与降级处理。新增按钮将直接复用这段逻辑，不引入第二套复制实现。

## 状态与错误处理

- 如果当前没有预览内容，复制按钮不会显示，因此不需要额外暴露新的空状态交互。
- 如果复制动作失败，沿用现有错误处理方式，在状态区显示错误信息。
- 抓取中应避免让用户误操作旧预览对应的复制按钮，因此在新一轮抓取开始时隐藏该按钮。

## 测试与验证

需要验证以下场景：

1. 初始打开弹窗时，看不到预览区复制按钮。
2. 点击主按钮并成功抓取后：
   - 预览区显示。
   - 新复制按钮显示在预览标题后面。
   - 主按钮仍会自动复制一次。
3. 点击新复制按钮后：
   - 不重新抓取。
   - 当前预览内容被再次复制。
   - 状态区提示复制成功。
4. 抓取失败时：
   - 预览区不显示。
   - 新复制按钮不显示。
5. 多次抓取、多次点击复制按钮时：
   - 始终复制当前最新预览内容。
   - 不出现旧内容残留带来的误导。

## 影响范围

本次改动只涉及弹窗 UI 与弹窗脚本，预计修改范围为：
- `popup.html`
- `popup.js`

不涉及：
- `manifest.json`
- `codex_credentials.js`
- 后端服务或外部接口协议

## 推荐实施方式

按最小改动原则实现：保留现有主流程，在预览区域追加一个次级复制入口，并复用现有复制能力。这样可以满足“再次复制”的需求，同时避免改变用户已经熟悉的主按钮行为。
